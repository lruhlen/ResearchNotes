Useful computer stuff figured out today:

  • A shell script called 'n' (in /usr/local/bin/. You'll need to use 'sudo' to do anything to the file.) that takes care of the setting up a growlnotification for any command you run. Use as follows:

    • > n command_name_here &

  • If you get the following error when attempting to use growlnotify:

    • growlnotify[681:903] could not find local GrowlApplicationBridgePathway, falling back to NSDNC

    • Go to System Preferences --> Growl --> Stop Growl (then Start Growl)

  • From last time: I copied over a clean version of Peter's code (as initially modified by Greg to get it to work) from my "backup" dir in to "UnalteredCode" in my working dir.

    • All runs involving Peter's version of the code mentioned here are produced using that clean copy of his code in my working dir. (I'm not messing with the actual backup copy!)

  • If you want to add a constant numerical value to the names of all the (numbered) files in a directory, enter the following command at the unix prompt, when you're within that directory:

    • ls *_model_info.txt | sort -r | awk -F'[_]' '{old=$0; n=$1+100; sub($1,n,$0); print "mv " old " "$0}' | sh

    • If you want a simulation to 'pick up' where the results of a previous one left off, make sure to set MODA = -1 in the .start file

  • Figured out how to get the legend on my python-generated figures to have multiple columns (and thereby fit on the visible portion of the graph):

    • legend(prop={'size':8},loc='lower center', ncol=3, fancybox=True, shadow=True)

    • Updated both MyPythonGUIPlottingScript.py and plot_HR_evolution.py to include this nicer, more legible version of the legend.

  • To set a pop-up notification on my desktop that lets me know when one of these really long thecode.f runs is finished (running in the background), start the run like this:

    • ./thecode < inputfile.start > outputfile && growlnotify -s -m 'TheCode finished running' &

So, you can't really run the code for 10000 timesteps and have it write output every 1000 steps. For now, my best work around is to let the code go for 9500 time steps, and have it write out model data every 500 steps.

My impression/summary of what the difference between all of these input files mean:

  1. All of the .start files are identical, with the trivial exception of the binary output filename.

  2. The main changes between a .start file and a .inp file for a given mass seem (to me) to be:

    1. The Smin and Smax values for the luminosity change a lot. I'm not sure what the effect of having a negative Smax value is, but my guess is those +/-2e37 values have something to do with relaxing the fitting/evolution constraints on the luminosity after you've gotten yourself a model converged from the sort-of-but-not-really physical n=3/2 polytropic starting conditions.

    2. The epsP/R/L/T convergence requirements get stricter once you move to the .inp file. (The change is the same for all three masses being modeled.)

    3. Atmx: These values stay within the same order of magnitude between the .start and .inp files, but they do change. In the 3Msun case, this value goes up a little, while it goes down by no more than ~1/2 in the other two cases.

    4. Atmn: These uniformly go down from 1e4 in the .start files to 5e3 in the .inp files. Probably because the starting polytropic model doesn't have an atmosphere, so... I have a vague notion that this has something to do with the outer boundary conditions, and with initially setting the coolest atm temperature to be something that the code can successfully converge into a more physical configuration from that polytropic input model. (Note: the 'surface' temperature of our own sun is ~ 5.5e3).

    5. dTax: "the maximum relative DELTAU in "atmos" normalized by TAU". This value always goes down from 1e-5 in the .start files to 1e-6 in the .inp files. I think this dTax parameter is basically controlling the resolution of the atmospheric calculations (in terms of optical depth). So, for the .inp (i.e., more evolved) produced models, we're wanting to keep better/more accurate track of what's going on in the atmosphere.

    6. dLmx: This is the maximum allowed jump in luminosity (between mass cells, I assume?), and its value gets a lot smaller once you hit the .inp files. I'm guessing that this causes the code to resolve the model's luminosity structure more finely. Not entirely sure why it needs to change so drastically between the .start and .inp files, though.

    7. dXmx: This always changes from 5e-2 to 4e-2 between the .start and .inp files. That doesn't seem like a big difference to me, but it does show up across all three masses, so maybe it's more important than I think it is.

    8. dPmx: This is the maximum allowed jump in pressure (between mass cells, right?), and its value decreases slightly between .start and .inp in the 1 and 3 Msun cases, though not in the 0.3Msun case. I don't think this is variable is what's causing the change in behavior between the .start and .inp cases (i.e. the kickoff of contraction with the .inp starting conditions).

    9. dPmn: This is the minimum allowed jump in pressure (between mass cells?), and its value increases by a factor of ten for all masses between the .start and .inp cases.

    10. dZdt: "relative speed with which the grid cell M(N-1) can move". I guess this controls how quickly the surface of the star can expand/contract? It increases between the .start and .inp cases by a factor of a few. The amount of increase is inversely proportional to the mass of the system being modeled. This parameter might be what allows the model to contract once you move into the .inp phase of the simulation.

    11. Chmx: Maximum allowed jump in chemical evolution (between mass cells?). It increase by a factor of ~2 between the .start and .inp cases (though less for the 3Msun case). This is probably increased in the .inp phase to allow the star to cope with the chemical evolution and layering produced by fusion. This should not be important for systems where fusion never really kicks off, such as Jupiters.

    12. That second abundance parameter (changes from -4e-4 to +4e-5). This is the deuterium abundance parameter. When it's set to a negative value (as in the .start files), the deuterium abundance is *not* updated after each timestep. I'm not sure why the deuterium abundance would be more stable over time in the young (.start) system, rather than in the more evolved system, because I thought deuterium burning is a more significant effect in young, pre-MS systems (and also in brown dwarfs-- basically, in any system where H-->He fusion hasn't kicked off yet).

  3. Of the changes listed above, I think (a) and (j) are the ones that most likely account for the change in the simulations' behavior (contracting vs. not) between the .start and .inp phases.

  4. I'll test this hunch out by making a .inp file for the 1Msun case that's identical to the .start file, except in the following respects:

    1. The dTime values will remain as they are in the existing .inp file

    2. The input file will be set to the binary model file

    3. Either the Smin, Smax, or dZdt values will be toggled between their .start and .inp values.

  5. Run each of those 3 .inp inputs on the output of pmsstar1.start, plot the results, and see which (if any) cause the simulation to start contracting the model.
