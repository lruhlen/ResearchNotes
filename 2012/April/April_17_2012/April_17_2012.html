  1. What do the various input parameters to TheCode.f mean (of the set that might have some effect on those sudden plunges in luminosity from last time). Stuff in quotes is copied verbatim from the documentation in pmsstar1.start. Other text is my personal notes/guesses.

    1. epsL:

      1. "Sets the convergence criterion for the luminosity variable."

      2. If the largest unscaled correction to any of the luminosity values is above this threshold, the code considers the model 'unconverged'.

      3. My sense is that this convergence parameter is related to the stability of the solution.

      4. Decreasing this value might help to prevent those luminosity flips, but it's not the first thing I'm planning try.

    2. SMIN:

      1. "...minimum values are added to the normalization of the corrections of "henyey" variables (prevents division by zero)"

      2. When the luminosity SMIN value is set to 1e10, the code throws an error whenever any of the luminosity values within the model go below that SMIN value.

      3. However, when the luminosity SMIN value is set to 2e37 (as it is in the .inp files), the code throws no errors, even though all of the luminosity values are much smaller than that.

        1. Based on what I see in TheCode.f, this is related to the fact that SMAX has been set to a negative value every time SMIN has been set to this very large, positive value. I'm not 100% sure of this, but having a negative SMAX may set the AFAC parameter to a negative value, and then... if/else conditions that skip you over the section of the code that takes you to the 'stop/error' exit from that subroutine... profit?!

      4. I need to look within the actual body of TheCode.f to see where SMIN pops up, because I still have only the haziest idea of precisely how it's enforced or used or whatever the proper verb for its function is.

        1. It looks like it only pops up in the derivative calculations in subroutine Henyey.

        2. Interestingly, based on the if/else structure of that part of the code, you can only get that 'stop: variable below limit' error if the SMIN value is positive. This is something to keep in mind the next time I find myself trying to get around that error...

    3. SMAX:

      1. "The following gives maximum relative changes to the "henyey" variables P, R, L, and T during each iteration. If > 0, then positive definite variables are assumed."

      2. I think this is the most likely culprit for the strange luminosity flips from last time. In runs where that behavior surface, SMAX (for the luminosity) was set to -2e37, which (as I understand it, at least) removes all restrictions on how much the luminosity is allowed to change in each iteration.

      3. I need to figure out what a reasonable value for SMAX (lum.) should be.

        1. By how much did the luminosity at the outer edge of the internal structural models change during each of the observed flips?

          • 0.2 Msun no fusion, step 49 flip: by ~0.002 Lsun, or 7.68e30 erg/s (from ~1e31 to ~4e30 erg/s) <-- maybe the reason the simulation didn't crash after this first hiccup; the change was less than a full order of magnitude.

          • 0.2 Msun no fusion, step 63 flip: from ~1e30 to ~1e28 erg/s <-- a change of 2 orders of magnitude might well be more 'fatal' to the simulation.

          • 0.1 Msun no fusion, penultimate vs. final recorded model (10 timesteps apart from each other): from ~3e31 erg/s to ~3e25 erg/s

        2. By how much did smoothly-decreasing lum. profiles change from step to step?

          • 0.2 Msun no fusion: The outer lum. value decreased by ~1e29 erg/s from one timestep to the next

          • 0.1 Msun no fusion: The outer lum. value decreased by ~1e31 erg/s every 10 timesteps --> ~1e30 erg/s from one timestep to the next.

        3. What was the luminosity of the polytropic input model?

          • 0.2 Msun with fusion (model 0): ~6.5e32 erg/s

          • 0.1 Msun with fusion (model 0): ~2.2e32 erg/s

        4. What was the luminosity of the first converged, with-fusion model?

          • 0.2 Msun with fusion (model 1): ~1.3e33 erg/s

          • 0.1 Msun with fusion (model 1): ~3.4e32 erg/s

        5. What was the luminosity of the first converged model in the no-fusion sequence?

          • 0.2 Msun no fusion (model 0): ~1.2e33

          • 0.1 Msun no fusion (model 0): ~3.4e32 erg/s

  2. Other input parameters that might have an effect on those luminosity 'jumps':

    1. dLmx and dLmn:

      1. "designate the maximum and minimum jumps in luminosity normalized by the maximum luminosity. Grid points will be automatically added if dLmx is exceeded, but deleted only if all conditions are met (there are also pressure, abundance, and grid separation criteria)."

      2. I should calculate and compare the dLdM values near the critical point in the 'jumpy' models vs. a 'smooth' models to see if there's any systematic difference between those two cases.

        1. My suspicion is that these parameters are not the main culprits, but it's plausible enough that I should do this analysis if the SMAX adjustments don't solve the problem.

    2. CHMN and CHMX:

      1. " are the minimum and maximum allowed relative changes of all henyey variables. If the relative changes are below CHMN the time step is increased (if allowed by dTMX), and if the relative changes are above CHMX the time step is decreased (if allowed by dTMN)."

      2. "relative changes" --> relative to what?

      3. These parameters only show up in the code once you've gotten a converged model. The 'change' value in TheCode.f (to which these parameters get compared) is just the sum of the absolute values of all the epsL/T/P/R variables; these, in turn, are just the largest changes applied to the dependent variables.

        1. I don't think changing the CHMN/MX values will have an immediate, dramatic impact on those luminosity 'flips.' I'd look into fixing the flip problem with epsL alone before going after these. These parameters are last on the list of things I'd tweak for this problem.

    3. dTIM/dTMX/dTMN:

      1. These are the timestep size parameters. They're basically coupled to the dLmx/mn parameters: if you decrease dTim, you'll almost certainly end up decreasing dL, too.

      2. I may play around with these, but I think that'd essentially end up being a less efficient way of regulating the effects of the dLMX/dLMN parameters.
